# UI/UX 개선 제안서

작성일: 2026-06-04  
대상: `blog_vue3_front` Vue 3 개인 워크스페이스 앱

## 1. 진단 요약

현재 프로젝트는 대시보드, 메모, Todo, 파일 스토리지, AI Chat을 하나의 개인 생산성 워크스페이스로 묶고 있습니다. 제품 방향은 명확합니다. 사용자는 한 화면에서 최근 작업을 확인하고, 메모를 작성하며, 파일을 올리고, 작업을 정리하고, AI와 대화할 수 있습니다.

다만 현재 UI는 "멋진 다크 글래스 인터페이스"의 인상이 강한 반면, 정보형 도구로서 필요한 가독성, 상태 피드백, 일관성, 오류 회복성이 부족합니다. 특히 실제 로컬 렌더링 확인 결과 첫 대시보드 화면은 본문 카드와 텍스트가 매우 어둡게 보여 사용자가 핵심 정보를 즉시 파악하기 어렵습니다.

핵심 개선 방향은 다음과 같습니다.

- 장식 레이어와 글래스 효과를 줄이고, 콘텐츠 대비와 정보 위계를 강화합니다.
- 화면별 헤더, 버튼, 빈 상태, 오류 상태, 저장 상태를 하나의 디자인 시스템으로 통합합니다.
- API 실패나 데이터 없음 상황에서도 화면이 비어 보이지 않게 안내와 재시도 액션을 제공합니다.
- 모바일에서는 하단 탭, 고정 헤더, 채팅 입력창, 모달이 서로 겹치지 않도록 실제 사용 흐름을 기준으로 재정렬합니다.
- 메모, Todo, 스토리지, 채팅 각각의 주요 작업에 "다음 행동"이 명확히 보이도록 워크플로를 다듬습니다.

## 2. 현재 UI/UX 강점

### 2.1 제품 구조가 이해하기 쉽다

사이드바와 모바일 하단 탭의 정보 구조는 Home, Memos, Tasks, Storage, Chat으로 단순합니다. 개인 워크스페이스 앱으로서 사용자가 기능 범위를 빠르게 이해할 수 있습니다.

### 2.2 기능별 기본 플로우가 이미 구현되어 있다

메모 목록/편집, Todo 빠른 추가/그룹 관리, 스토리지 업로드/다운로드/삭제, 채팅 세션 관리가 분리되어 있습니다. 새 기능을 대규모로 추가하기보다 기존 플로우의 완성도를 높이는 것이 효과적입니다.

### 2.3 디자인 토큰 기반의 확장 가능성이 있다

`variables.css`에 색상, 간격, 폰트 크기, radius, 레이아웃 토큰이 정의되어 있어 개선 작업을 중앙에서 시작할 수 있습니다. 이 기반은 유지하되 실제 컴포넌트와 화면에서 일관되게 쓰도록 정리해야 합니다.

## 3. 주요 문제점

### 3.1 첫 화면 가독성이 낮다

첫 화면의 장식 레이어와 반투명 글래스 표면이 겹치면서 본문 카드, KPI, 최근 활동 영역이 배경과 섞여 거의 보이지 않는 문제가 확인되었습니다.

개선안:

- 장식 레이어를 제거하고, 주요 콘텐츠 영역 뒤에는 불투명 표면을 사용합니다.
- 대시보드 카드의 기본 배경을 `rgba(24, 24, 27, 0.88)` 이상으로 올립니다.
- 첫 화면에서는 장식보다 Today Brief, Quick Clipboard, KPI 값이 먼저 읽히도록 대비를 재조정합니다.
- `prefers-reduced-motion` 환경에서도 주요 화면 전환만 최소한으로 유지합니다.

우선순위: P0

### 3.2 전역 타이포그래피가 지나치게 조여져 있다

`global.css`에서 `body` 전체에 `letter-spacing: var(--tracking-tight)`가 적용되어 있고, 헤딩에는 별도 음수 자간도 적용됩니다. 정보형 UI에서 본문, 버튼, 작은 캡션까지 모두 좁아지면 한글/영문 혼합 텍스트의 가독성이 떨어집니다.

개선안:

- 전역 `body`의 letter-spacing은 `0`으로 되돌립니다.
- 큰 헤딩에만 제한적으로 타이트한 자간을 적용합니다.
- 캡션, 버튼, 입력 필드는 자간을 기본값으로 유지합니다.

우선순위: P1

### 3.3 디자인 시스템 컴포넌트가 중복되어 있다

Base 컴포넌트가 `shared/ui`, `components/ui`, `components/base`, `components/layout`에 중복 존재합니다. `shared/ui/BaseButton.vue`는 다시 `components/ui/BaseButton.vue`를 감싸는 구조이고, 카드도 동일한 래퍼 구조를 갖습니다. 이 구조는 버튼 variant, hover, loading, focus, radius가 화면마다 다르게 보일 가능성을 높입니다.

개선안:

- 최종 사용 경로를 `src/shared/ui` 하나로 통일합니다.
- 기존 `components/ui`, `components/base`, `components/layout`은 점진적으로 deprecated 처리 후 제거합니다.
- 버튼, 아이콘 버튼, 카드, 입력, 모달, 스켈레톤의 상태별 스펙을 문서화합니다.
- 아이콘 버튼은 공통 `IconButton` 컴포넌트로 분리하고 `aria-label`, tooltip, loading, disabled를 표준화합니다.

우선순위: P1

### 3.4 API 실패 시 본문이 비어 보인다

로컬 실행에서 API 데이터가 없는 상태로 `/memo`, `/todo`, `/s3` 라우트를 열면 본문 텍스트가 거의 드러나지 않고, 사용자는 화면이 고장났는지 빈 데이터인지 구분하기 어렵습니다. 일부 composable은 실패를 `console.error`로만 처리하고 화면 상태로 노출하지 않습니다.

개선안:

- 모든 데이터 화면에 `loading`, `empty`, `error`, `success` 상태를 명시합니다.
- 오류 상태에는 "다시 시도", "설정 확인", "로컬 API 연결 필요" 같은 다음 행동을 제공합니다.
- 대시보드는 각 카드별 로딩 실패를 전체 화면 실패로 만들지 않고 부분 실패로 표시합니다.
- `useAsync` 패턴을 메모, Todo, 스토리지, 대시보드에 일관되게 적용합니다.

우선순위: P0

### 3.5 액션의 의미와 위험도가 시각적으로 충분히 구분되지 않는다

저장, 새로고침, 삭제, 복사, 다운로드, 설정이 모두 작은 아이콘 버튼으로 표시되는 화면이 많습니다. 빠른 사용에는 좋지만, 위험 액션인 삭제와 상태 변경 액션인 저장의 중요도가 충분히 구분되지 않습니다.

개선안:

- 주요 CTA는 텍스트가 있는 버튼으로 유지합니다. 예: `New Memo`, `Save Changes`, `Upload File`.
- 반복 도구 액션은 아이콘 버튼으로 두되 tooltip과 `aria-label`을 필수화합니다.
- 삭제는 hover에서만 보이지 않게 하지 말고, 카드 focus/keyboard 접근 시에도 접근 가능하게 합니다.
- 저장되지 않은 변경이 있을 때는 헤더에 명확한 상태 배지와 저장 버튼 강조를 제공합니다.

우선순위: P1

### 3.6 모바일 레이아웃의 충돌 위험이 있다

앱 전체에는 모바일 하단 탭이 있고, 채팅 화면에는 별도 모바일 헤더와 채팅 composer가 있습니다. `Chat.vue`는 모바일에서 `height: calc(100dvh - 92px - safe-area)`를 사용하고, 전체 레이아웃도 하단 탭 공간을 별도로 둡니다. 작은 화면에서 고정 영역이 많아 입력창, drawer, 하단 탭이 서로 압박할 수 있습니다.

개선안:

- 모바일 레이아웃 토큰을 `--mobile-nav-height`, `--mobile-content-bottom`, `--mobile-header-height`로 분리하고 실제 컴포넌트 높이와 일치시킵니다.
- Chat 화면에서는 하단 탭이 필요한지 재검토합니다. 채팅 집중 모드에서는 탭을 축소하거나 composer 위 공간을 보장합니다.
- 모달은 모바일에서 full-width bottom sheet 패턴을 사용합니다.
- 360px, 390px, 430px, 768px, 1024px 기준 스냅샷 테스트를 추가합니다.

우선순위: P1

## 4. 화면별 개선 제안

### 4.1 Home Dashboard

현재 역할: 오늘 요약, 클립보드, KPI, 최근 활동을 보여주는 허브.

문제:

- 첫 화면 본문 대비가 낮아 핵심 정보가 읽히지 않습니다.
- `Welcome Back, deepria`는 개인화 느낌은 좋지만, 실사용 도구의 첫 정보로는 다소 정적입니다.
- Quick Clipboard가 어떤 저장소와 연결되는지, 저장 후 어디서 찾는지 명확하지 않습니다.
- 최근 활동이 없을 때의 안내가 약합니다.

개선안:

- 상단 카드 제목을 "Today Brief"보다 실제 요약 중심으로 바꿉니다. 예: `오늘 할 일 0개, 메모 0개`.
- KPI 카드는 클릭 가능한 카드임을 hover뿐 아니라 chevron 또는 보조 텍스트로 표시합니다.
- Quick Clipboard는 `Save`, `Copy`, `Clear`, `Refresh` 액션을 그룹화하고 마지막 저장 시간을 표시합니다.
- Recent Activity 빈 상태에는 "메모 작성", "파일 업로드", "작업 추가" 빠른 액션을 제공합니다.
- 장식 레이어 없이 대시보드 카드의 콘텐츠 대비를 우선합니다.

### 4.2 Memos

현재 역할: 메모 검색, 목록, 미리보기, 편집 진입.

문제:

- 검색 결과 없음과 메모 없음이 같은 empty state로 보입니다.
- 메모 카드 삭제 버튼이 hover에 의존합니다.
- 미리보기가 카드 아래로 확장되는 방식은 그리드 레이아웃에서 주변 카드 흐름을 크게 흔들 수 있습니다.
- 편집 화면에서 저장 후 무조건 목록으로 이동합니다. 사용자는 저장 결과를 확인하거나 이어서 편집하고 싶을 수 있습니다.

개선안:

- empty state를 `아직 메모 없음`과 `검색 결과 없음`으로 분리합니다.
- 카드 클릭은 preview, 더블클릭 또는 명시 버튼은 edit로 역할을 분리합니다.
- 미리보기는 오른쪽 drawer 또는 하단 detail panel로 통일해 그리드 흔들림을 줄입니다.
- 편집기에는 `Saved just now`, `Unsaved changes`, `Save failed` 상태를 표시합니다.
- 새 메모 저장 후에는 목록 이동 대신 현재 편집 화면에 머물고 `Back to Memos`를 제공합니다.

### 4.3 Tasks

현재 역할: 빠른 작업 추가, 그룹/우선순위 관리, 작업 목록 편집.

문제:

- 설정 모달 안에서 작업 편집과 그룹/우선순위 관리가 섞여 인지 부담이 큽니다.
- 저장되지 않은 변경은 버튼 pulse에 의존해 의미가 직관적이지 않을 수 있습니다.
- Todo 목록의 그룹 헤더는 있으나 전체 진행률, 필터, 완료/미완료 토글이 부족합니다.

개선안:

- 작업 편집 모달과 그룹/우선순위 관리 모달을 분리합니다.
- 헤더에 `Unsaved changes` 배지와 `Save Changes` 텍스트 버튼을 제공합니다.
- 상단에 `All`, `Active`, `Done` segmented control을 추가합니다.
- 그룹별 작업 수와 완료율을 표시합니다.
- 빠른 추가 입력 후 생성된 항목에 짧은 highlight를 줍니다.

### 4.4 File Storage

현재 역할: 파일 업로드, 파일 목록, 미리보기/다운로드/삭제, 비밀번호 보호.

문제:

- 업로드 패널과 파일 목록의 관계는 좋지만, 보호 키가 어떤 액션에 필요한지 사전 설명이 부족합니다.
- 비밀번호 입력 모달이 preview/download/delete 모두에 재사용되지만, 현재 하려는 액션 맥락이 제목에 반영되지 않습니다.
- 이미지 외 파일 preview의 fallback이 약합니다.

개선안:

- 업로드 모달에 `Optional protection key` 설명을 짧게 추가합니다.
- 보안 모달 제목을 `Preview protected file`, `Download protected file`, `Delete protected file`처럼 액션별로 바꿉니다.
- 파일 타입별 아이콘, 크기, 업로드 날짜, 보호 여부를 한 줄 메타로 표준화합니다.
- 미리보기 불가 파일에는 다운로드 CTA와 파일 정보를 보여줍니다.
- 업로드 진행률은 실제 progress 이벤트와 연결하거나, 불확실한 경우 indeterminate 상태로 표시합니다.

### 4.5 Chat

현재 역할: AI Assistant와 세션 기반 대화.

문제:

- 데스크톱에서 앱 전체 사이드바와 채팅 세션 사이드바가 나란히 있어 왼쪽 영역이 넓게 소모됩니다.
- `Online` 상태가 실제 모델/API 연결 상태를 보증하는지 불명확합니다.
- 빈 상태가 친근하지만, 사용자가 바로 시작할 수 있는 제안 프롬프트가 없습니다.
- 채팅 composer가 하단에 고정되어 있지만 모바일 하단 탭과 공간 충돌 위험이 있습니다.

개선안:

- Chat 화면에서는 전역 사이드바를 접을 수 있게 하거나 채팅 세션 사이드바 폭을 줄입니다.
- `Online` 대신 `Ready`, `Connecting`, `Error` 등 실제 상태 기반 라벨을 사용합니다.
- 빈 상태에 3개의 quick prompt를 제공합니다. 예: `오늘 작업 정리해줘`, `메모 초안 만들어줘`, `파일명 정리 규칙 제안해줘`.
- 사용자 메시지는 카드형, assistant 메시지는 문서형으로 유지하되 코드블록 복사 버튼을 추가합니다.
- 장문 대화에서 "맨 아래로 이동" 플로팅 버튼을 제공합니다.

## 5. 디자인 시스템 개선안

### 5.1 색상과 대비

권장 원칙:

- 정보 카드 기본 배경은 앱 배경과 분리될 만큼 충분히 불투명해야 합니다.
- 본문 텍스트 대비는 WCAG AA 수준을 목표로 합니다.
- primary 색상 `#5EEAD4`는 강조/상태/포커스에 사용하고, 일반 텍스트 버튼의 배경으로 남용하지 않습니다.
- 위험 색상은 삭제/파괴 액션에만 씁니다.

권장 토큰:

```css
--color-bg-base: #09090B;
--color-bg-surface: #18181B;
--color-bg-card: rgba(24, 24, 27, 0.92);
--color-bg-card-muted: rgba(24, 24, 27, 0.82);
--color-text-primary: #FAFAFA;
--color-text-secondary: #D4D4D8;
--color-text-muted: #A1A1AA;
```

### 5.2 컴포넌트 표준화

우선 통합할 컴포넌트:

- `BaseButton`: primary, secondary, ghost, danger, icon
- `BaseIconButton`: tooltip, aria-label, size, variant, loading
- `BaseCard`: default, elevated, interactive, selected
- `BaseEmptyState`: title, description, primaryAction, secondaryAction
- `BaseErrorState`: title, description, retryAction, details
- `BasePageHeader`: title, subtitle, actions, status
- `BaseToolbar`: search, filter, sort, view mode

### 5.3 모션

권장 원칙:

- 화면 진입 모션은 150-200ms 안에서 끝냅니다.
- 콘텐츠 우선 화면에서는 시각 노이즈가 생기는 장식 모션을 사용하지 않습니다.
- `prefers-reduced-motion: reduce`에서는 transition을 최소화합니다.
- hover scale은 카드보다는 버튼/작은 인터랙션에 제한합니다.

## 6. 정보 구조와 문구 개선

현재는 영어와 한국어가 혼합되어 있습니다. 예를 들어 대시보드와 메모는 영어 중심이고, 채팅 빈 상태는 한국어입니다. 사용자의 주 사용 언어가 한국어라면 전반 문구를 한국어로 통일하는 것을 권장합니다.

권장 용어:

- Home → 홈
- Memos → 메모
- Tasks → 할 일
- Storage → 파일
- Chat → AI
- Today Brief → 오늘 요약
- Quick Clipboard → 빠른 클립보드
- Recent Activity → 최근 활동
- Save Changes → 변경사항 저장

단, 개발자 도구 성격을 살리고 싶다면 내비게이션은 짧은 영어로 유지하고, 빈 상태/오류/도움말 문구만 한국어로 통일해도 됩니다.

## 7. 접근성 개선

필수 개선:

- 모든 아이콘 버튼에 `aria-label`을 제공합니다.
- hover에만 노출되는 삭제 버튼은 키보드 focus에서도 노출합니다.
- 모달은 열릴 때 첫 입력 요소로 focus를 이동하고, 닫힐 때 트리거로 focus를 복귀합니다.
- Toast 메시지만으로 성공/실패를 전달하지 말고 화면 상태도 함께 갱신합니다.
- 색상만으로 그룹/우선순위를 구분하지 않고 텍스트 라벨을 함께 제공합니다.
- 채팅 메시지 영역은 새 assistant 응답을 `aria-live`로 적절히 알립니다.

4. Home 대시보드 카드 배경을 더 불투명하게 조정합니다.

7. `Online` 배지는 실제 연결 상태를 반영할 수 있을 때만 사용합니다.

## 11. 결론

이 프로젝트는 기능 범위보다 UI/UX 완성도가 병목입니다. 현재 구현된 기능들을 그대로 살리면서, 첫 화면 가독성, 상태 피드백, 디자인 시스템 통합, 모바일 충돌 방지만 잡아도 제품 인상이 크게 좋아질 수 있습니다.

가장 먼저 해야 할 일은 장식 레이어와 글래스 강도를 낮춰 "보이는 앱"으로 만드는 것입니다. 그 다음 빈 상태와 오류 상태를 정리하고, 버튼/카드/입력의 표준 컴포넌트를 통합하면 전체 앱이 훨씬 안정적이고 전문적인 개인 워크스페이스로 느껴질 것입니다.
